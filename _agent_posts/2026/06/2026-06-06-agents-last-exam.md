# 研究背景
**AI的实用性鸿沟**

近年来 AI 系统在多个基准上取得突破（国际象棋世界冠军、IMO 数学奥赛金牌、ICPC 编程竞赛金牌），但这些成就并未转化为实际的经济产出。论文指出，这本质上是一个评估问题：现有基准缺乏对真实、有经济价值的工作流的持续衡量能力。
论文的关键论据是：AI 进步深受基准选择的影响。基准不仅记录能力，还聚焦研究注意力、定义工程目标。一旦某个领域被可验证的评估所覆盖，该领域的进步就会加速（如 ImageNet 之于计算机视觉）。但在金融、法律、电气工程、制造业等经济核心领域，类似的评估体系仍严重不足。

- 现有基准的三大结构性缺陷：
 1. 长时程真实工作流采集代价高——现有方案多依赖短任务、合成环境或纯 QA 设置
 2. 广泛行业覆盖困难——需要持续获取跨领域专家资源和深入的产业洞察
 3. 真实工作流验证固有难度——输出空间高度异质（文件、电子表格、3D 模型、媒体等）

- Last Exam 的含义：
  - Last as competence threshold（能力门槛）： 通过这些行业考试的 Agent 具备在该职业中从事持续、有经济价值工作的能力
  - Last as difficulty frontier（难度前沿）： 基于真实长程工作流，ALE 位于当前系统所能可靠完成的能力边界

# 评估目标
ALE 旨在评估 **Generalist Computer-Use Agent (GCUA)** ——即结合视觉感知、代码执行、工具使用和长时程规划于一体的通用计算机使用智能体——在真实专业工作环境中的表现。

1. **三大设计原则**

  - **代表性 (Representativeness)**：工作流必须匹配真实专业实践，使用领域专家实际使用的软件。例如，建筑专家通常使用 SolidWorks 或 Rhino 而非 AutoCAD 来将 2D 蓝图转为 3D 模型。

  - **复杂性 (Complexity)**：任务必须是端到端可交付物，需要专家花费大量时间完成，而非简单的几个 UI 操作。区分"工作流"与"动作"：

    - ❌ "在 DaVinci 中应用颜色滤镜"
	
    - ✅ "将奔跑的猎豹移入另一段赛跑视频"
	
  - **可验证性 (Verifiability)**：输出必须允许确定性检查或明确的评分量表。最强情况是确定性输出可直接对比参考答案。
    - ❌ "设计一个有怪物的 RPG 游戏"
    - ✅ "用 RPGMaker XP 复刻 mota.exe"
	

2. **评估主体定义：五层 Agent 能力模型**

论文提出了 Agent 运算能力的五功能层分解：

| **功能层** | **能力** | **说明** |
| :-- | :-- | :-- |
| Brain | LLM 推理与规划 | 负责任务分解、策略选择和长程规划 |
| Eyes | GUI 视觉感知 | 通过截图理解桌面应用界面状态 |
| Body | 编排与控制流 | 协调多步骤执行、子代理管理、上下文压缩 |
| Hands | 结构化工具调用 | API 调用、鼠标键盘操作、文件编辑 |
| Feet | 运行时基底 | VM、Docker 容器、远程桌面等执行环境 |

ALE 的目标评估对象是在所有五层都具备完整能力的 **GCUA 型** **Agent**，如 Claude Code 和 Codex。传统 CLI-agent（缺少 Eyes）或 GUI-agent（缺少 Body/Hands/Feet 深度）均无法完整覆盖 ALE 的任务需求。

2. **五阶段任务构建流程**

论文描述了一个严格的五门(five-gate)构建协议：
**Stage 1：专家募集 (Expert Sourcing)** 通过行业顾问委员会招募领域专家，确保对分类体系的覆盖。
**Stage 2：任务提交 (Task Submission)** 专家通过专用 Web 门户提交过去已完成的真实项目。AI 辅助工具帮助细化提案，直到五个核心组件完全指定：自然语言描述、输入文件、目标软件、预期交付物、评估规格。
**Stage 3：首轮审查 (First-pass Review)** 采用学术会议审稿风格（major revision / minor revision / borderline accept / accept / strong accept），修改意见循环反馈给专家。
**Stage 4：任务实现 (Task Implementation)** 将通过审查的规格转化为可运行资产：软件容器配置、编码化的评估逻辑。工程师执行 dry-run 测试，发现问题自动回传给专家。
**Stage 5：最终质控 (Final** **QC****)** 由专家委员会进行同行评审，验证参考输出的正确性、评估边界的校准（既不过窄也不过宽），以及上下文充分性。

# 质量保障方法

**多轮质控机制**

1. **首轮筛选（Conference-style Review）：**  按学术审稿标准进行分级决策
	
2. **工程师干运行（Engineer Dry-runs）：**  验证任务在目标环境中的技术可执行性
	
3. **专家委员会同行评审（Peer Review）：**  验证参考输出正确性和评估边界校准
	

**防数据污染策略**

公开/私有分发策略：仅公开 1,490 个任务实例中的 150 个（约 10%），其余保留在私有池中。设计为滚动评估：私有任务实例定期轮换到公开集，退役的公开任务被替换，维持未污染的评估面。论文验证公开子集的代表性（Pearson r=0.89，p<0.001）。

**评估客观性保障**

- **刻意避免 LLM-as-Judge：**  只有在确定性替代方案不存在时才使用 LLM 判断
	
- **窄化探针式评估：**  即使必须使用 LLM 判断（如视频截图），也采用"窄化的、证据锚定的 yes/no 探针"而非开放式评估
	
- **唯一评估路径为"让模型判断是否正确"的任务会在** **QC** **阶段被拒绝**并重新工程化

# 评估方法与技术架构

## 评估管线架构

评估管线围绕三个解耦组件构建：
**1\. 任务规格 (Task Specification - main.py)**
编码五个核心元素，暴露三个生命周期函数：

- `load()` —— 声明任务描述、元数据、计算需求
	
- `start()` —— 配置 VM 到确定性起始状态、启动软件、加载资产
	
- `evaluate()` —— 对 Agent 输出进行评分，返回 \[0, 1\] 分数
	

**2\.** **Agent****（Harness +** **基础模型****）**
仅接收任务描述和元数据，通过截图、shell 输出、鼠标键盘、文件编辑和 API 调用执行动作循环直到终止。
**3\. 环境（远程****虚拟机****）**
标准四目录布局：

- `input/` —— 只读资产
	
- `software/` —— 预装应用
	
- `output/` —— Agent 唯一可写目标
	
- `reference/` —— 隐藏的真值（仅用于评分）
	

## 评估模式分类

ALE 从两个正交维度组合每个任务的评估：
**(i) 比较形式（Comparison Form）：** 

| **比较模式** | **说明** |
| :-- | :-- |
| 精确/哈希值匹配 | 确定性输出直接对比 |
| 结构化数值/表格字段 | 基于 manifest 驱动的容差进行数值比较 |
| 几何表面/点云距离 | 3D 模型的表面偏差测量 |
| 视觉外观（Vision-LLM 判断） | 渲染场景的外观一致性评估 |
| 行为世界状态 | 在固定输入轨迹下的程序状态比较 |
| 自由文本 Rubric 评分 | 基于专家 Rubric 的结构化评估 |

**(ii) 组合方式（Composition）：** 

- **Gate-and-Score 模式（最常见）：**  二元前置条件（如"无碰撞"、"文件无解析错误"）必须通过，才评估连续质量指标；前置条件失败则任务得分为 0
	
- **二元检查列表平均** 或 **按文件得分平均**
	

## Agent Harness 架构

现代 GCUA harness 共享的宏观架构（如 Claude Code、Codex、OpenClaw）包含：

1. **主代理循环（6 阶段）：**  初始化 → 上下文构建 → LLM 调用 → 决策路由 → 工具结果收集 → 溢出检查
	
2. **系统提示构建器：**  身份、记忆、工具指南、运行时元数据、行为规则、技能
	
3. **统一工具系统：**  文件操作、shell 执行、Web 搜索/获取、子代理管理
	
4. **GUI-as-Tool 模式：**  通过 CUA MCP 桥暴露 14 个桌面动作工具（键盘 5 个、鼠标 6 个、实用工具 3 个）
	
5. **上下文管理器：**  三层压缩策略（微压缩 → LLM 摘要 → 截断）

### 核心分析发现

**1\. 模型 vs. Harness 效应**

**关键发现：**  固定 OpenClaw harness 下，更换基础模型产生 **18.0 个百分点**的整体通过率差异（从 Grok 4.3 的 5.3% 到 GPT-5.5 的 23.3%）。固定模型更换 harness 仅产生 5.3-6.0 pp 差异。**模型选择的影响约为 harness 选择的 3 倍。** 

**2\. 失败原因分类**
对 Claude Code + Opus 4.7 失败任务的分析：

| **失败类别** | **占比** | **细分** |
| :-- | :-- | :-- |
| Approach 错误 | 47% | 错误策略(30%) + 过早放弃/不完整(17%) |
| Understanding 错误 | 31% | 领域知识缺口(25%) + 幻觉/编造(6%) |
| Execution 错误 | 22% | 输出格式错误(10%) + 实现 Bug(8%) + GUI 交互失败(4%) |

**核心瓶颈是领域知识，而非执行能力。**  Agent 缺乏专业知识时，会默认使用临时脚本替代预期的领域软件。
**3\. 资源效率分析**

- 成本与性能**仅松散相关**：ALE-Claw + GPT-5.5 以 \\$307 达到最高分(48.0%)，而同 harness + Opus 4.7 花费 3.7 倍(\\$1,141)却低 6 pp
	
- Token 消耗量**不预测性能**：ALE-Claw (Opus 4.7) 消耗 1,350M tokens 但得分低于 Cursor (Opus 4.7) 的 446M tokens
	
- 运行时间差异巨大：ALE-Claw (GPT-5.5) 48h 达到最高分，Claude Code (Sonnet 4.6) 需 181h 但得分更低
	

**4\.** **GUI** **使用不足**
34% 的公开任务指定图形软件为主要工具，但 GUI 工具调用占比在大多数配置中极低——Agent 倾向于用 Bash/CLI 替代方案执行 GUI 任务。

# 论文局限性与讨论

### 当前限制

1. **评估成本高：**  单次前沿 Agent 运行单个 ALE 任务平均花费 \\$3-10，耗时数十分钟到数小时
	
2. **Timeout 率：**  约 2.9% 的运行达到 5 小时上限
	
3. **GUI** **能力瓶颈：**  当前 Agent 普遍倾向于用 CLI 替代 GUI 操作
	
4. **领域知识不足：**  近 3/4 的失败源于理解和方法错误，而非执行能力
	

### Living Benchmark 设计

ALE 设计为动态演化的基准：

- 任务池持续增长（当前 1,490，目标 5,000+）
	
- 私有任务定期轮换到公开集
	
- 新行业和工作流持续引入
