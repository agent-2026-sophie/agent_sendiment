## 1\. MLE-bench 是什么

在评估 AI Agent 的机器学习工程能力时，MLE-bench 提供了一个观察真实进展并允许与人类水平直接对比的重要标尺。作为 OpenAI 的 Preparedness 团队于 2024 年 10 月发布的一项评估基准，MLE-bench 旨在衡量自主 AI Agent 执行真实机器学习工程任务的表现[\[1\]](https://arxiv.org/pdf/2410.07095)[\[2\]](https://aiwiki.ai/wiki/mle_bench)。与以往仅测试孤立代码技能的评测不同，MLE-bench 重点考察模型全盘处理复杂 ML 任务的综合能力，如数据集准备、模型训练、实验运行以及调试等[\[3\]](https://www.marktechpost.com/2024/10/12/openai-researchers-introduce-mle-bench-a-new-benchmark-for-measuring-how-well-ai-agents-perform-at-machine-learning-engineering/)。因此，该基准被广泛用于 Agent 性能对比、资源利用及稳定性研究，并可作为机器学习工程教育和开发测试的重要工具[\[4\]](https://blog.csdn.net/gitblog_00900/article/details/147067579)。

### 1.1 任务设计与评测环境

为保证测试任务的真实性，MLE-bench 精选了 75 个来自 Kaggle 的机器学习竞赛任务，涵盖表格数据、图像、自然语言处理和音频等多种模态[\[5\]](https://arxiv.org/abs/2410.07095)。++每一个任务都被设计为一个++++端到端++++的机器学习工程难题，要求++ ++Agent++ ++独立完成阅读竞赛描述、数据准备、++++模型训练++++、问题调试、策略迭代直至最终提交预测结果（++++CSV++++文件）的完整流程++[\[2\]](https://aiwiki.ai/wiki/mle_bench)。
这种端到端的设计需要在极为贴近真实场景的环境中运行。为此，MLE-bench 为每个 Agent 配置了基于 Ubuntu 20.04 的 Docker 容器运行环境，并提供高达 36 个 vCPU、440 GB 内存、4 TB 固态硬盘，以及一张 24 GB 显存的 NVIDIA A10 GPU[\[2\]](https://aiwiki.ai/wiki/mle_bench)。在每项竞赛测试中，Agent 拥有最多 24 小时的运行时间上限来产出最终提交结果[\[6\]](https://arxiv.org/pdf/2410.07095v1)。
在评分机制方面，MLE-bench 极具创新性地直接使用真实 Kaggle 比赛隐藏测试集的金、银、铜牌分数线对 Agent 提交的结果进行评分，从而使得 AI 与人类 Kaggler 的水平能够直接量化对比[\[7\]](https://openreview.net/forum?id=6s5uXNWGIh&noteId=hrgDj6lZaW)。

### 1.2 核心性能特征与局限性

从公开的评测结果看，Agent 在 MLE-bench 中的表现具备一定潜力且能通过重试策略实现性能倍增。例如，OpenAI 的 o1-preview 配合为 Kaggle 竞赛量身打造的 AIDE 框架，在单次运行中平均有 16.9% 的比赛能获得奖牌[\[1\]](https://arxiv.org/pdf/2410.07095)。而当给予多次尝试机会（如 pass@8）时，该模型获奖牌的比例会翻倍至 34.1%[\[8\]](https://openreview.net/pdf?id=6s5uXNWGIh)。更有数据显示，o1-preview 在评测中平均斩获 7 枚金牌，这一数字已超过成为 “Kaggle Grandmaster（Kaggle 大师）”所需的 5 枚金牌标准 (中等可信度)[\[9\]](https://m.thepaper.cn/newsDetail_forward_28996267?commTag=true)。
然而，这种前沿测试基准也揭示了一些明显的限制与挑战。首先是数据污染与剽窃风险，因为 Kaggle 比赛天然是公开的，如果模型在训练阶段“记住”了比赛的公开解法或高分笔记，测试结果就会被虚假抬高[\[10\]](https://arxiv.org/pdf/2410.07095?utm_source=www.therundown.ai&utm_medium=referral&utm_campaign=elon-musk-reveals-driverless-robotaxi)。部分竞赛还存在测试集数据泄露的问题，导致比赛变得过于简单，甚至不适合作为严谨的评测用例[\[11\]](https://deepwiki.com/openai/mle-bench/11-known-issues-and-troubleshooting)。其次，Agent 在复杂流程中依然表现出常见的失败模式，例如陷入调试死胡同、难以及时从错误步骤中恢复、无法处理某些奇异的数据格式，以及生成的提交文件格式不规范等[\[12\]](https://coale.science/p/6379e1c5-1f58-4802-bbd7-8c753cd9d693)。

## 2\. 和现有评测差在哪

与其他主流 AI Agent 评测框架相比，MLE-bench 最突出的差异在于**其任务的长周期性与高度“****端到端****”的解决闭环**。这使得它能测出模型在面临真实复杂系统工程时的潜力，同时也带来了一些适用边界。

### 2.1 任务真实性与端到端差异

传统的“一发式”代码生成基准（如 HumanEval）往往只测试大模型生成孤立代码块的能力，而忽略了阅读庞大代码库、理解长上下文、代码迭代与缺陷调试等关键技能[\[13\]](https://arxiv.org/pdf/2410.07095v3.pdf)。与这些基准相比，MLE-bench 需要 Agent 在真实 GPU 容器内完成全周期任务；与 ML-Bench 相比，后者偏向于考察 Agent 对既有代码库的理解与应用能力，而 MLE-bench 则是让模型面对开放性问题“从零开始”构建完整的机器学习解决方案[\[10\]](https://arxiv.org/pdf/2410.07095?utm_source=www.therundown.ai&utm_medium=referral&utm_campaign=elon-musk-reveals-driverless-robotaxi)。
MLE-bench 与目前代表性的 AI 评测基准存在不同的定位侧重点：

- **SWE-bench**：目前代码修复领域最权威的评测，侧重于软件工程的 Bug 修复，要求 Agent 消化 GitHub 真实 Issue 并生成拉取请求 (Pull Request)[\[14\]](https://blog.csdn.net/weixin_43857576/article/details/160632918)。
	
- **GAIA**：评估通用 AI 助手跨领域推理、多模态处理和多工具混合编排能力的基准[\[15\]](https://www.adaline.ai/blog/evaluating-ai-agents-in-2026)。
	
- **OSWorld**：针对真实操作系统的全栈计算机控制测试，要求模型在真实的 Ubuntu 虚拟机中操作浏览器、办公软件及进行系统级文件管理[\[16\]](https://benchmarkingagents.com/osworld/)。
	
- **AgentBench**：关注广度层面的测试，旨在评估 Agent 在操作系统、数据库等八种不同底层环境中的能力表现[\[17\]](https://www.marktechpost.com/2026/04/26/top-7-benchmarks-that-actually-matter-for-agentic-reasoning-in-large-language-models/)。
	

### 2.2 评估能力的适配与盲区

借助与人类直接对标的 Kaggle 奖牌体系，MLE-bench 极大地便利了对 AI Agent 数据科学建模水平与分数优化能力的直观评估[\[18\]](https://0h-n0.github.io/posts/paper-2410-07095/)。在顶尖模型架构与特定工具的组合下，例如 Gemini-2.0-Flash 配合 MLE-STAR 框架，Agent 已经可以在 44% 的 MLE-bench 任务中拿下奖牌，大幅超越了早期 AIDE 架构的水平[\[19\]](https://www.emergentmind.com/topics/mle-bench)。
不过，由于其将评测完全框定于“比赛分数优化”的场景中，MLE-bench 的覆盖面必然存在局限。它更适合回答“当前模型能否写出并训练出一个准确率足够高的算法”，而不适合回答现实业务中的诸多工程痛点。例如，它无法测试模型在生产环境下的部署能力、线上监控表现、模型的商业解释性、对复杂业务合规需求的理解，以及监管要求等实务机器学习工程中极其关键的环节[\[18\]](https://0h-n0.github.io/posts/paper-2410-07095/)。此外，MLE-bench 还揭示了 Agent 存在的泛化鸿沟（Generalization gap）问题，由于 Agent 经常在验证集与实际测试集表现之间存在 9% 到 13% 的偏差，这说明它们仍缺乏人类资深数据科学家所具备的稳健模型选择技能[\[2\]](https://aiwiki.ai/wiki/mle_bench)。
